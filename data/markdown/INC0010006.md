# INC0010006

**Short Description:** GitLab CI/CD runners unresponsive — all pipelines queued
**Priority:** 3
**State:** 7

## Description
All 8 GitLab CI/CD shared runners are showing as 'offline' in the GitLab admin panel. 120+ pipelines are stuck in 'pending' state. Development velocity is completely halted. Issue started at approximately 14:30 UTC.

## Resolution
ROOT CAUSE ANALYSIS:
The gitlab-runner service on all 8 runner VMs stopped responding after an automatic OS security patch (applied via RHEL Satellite at 14:15 UTC) updated the 'nftables' package, which reset firewall rules and blocked outbound connections from the runners to the GitLab server on port 443.

RESOLUTION STEPS TAKEN:
1. Checked runner status in GitLab Admin → Runners panel — all showed last contact    timestamp of 14:16 UTC, correlating with the patch window.
2. SSH'd into runner VMs and checked gitlab-runner service:
   systemctl status gitlab-runner — service was running but logs showed TLS    connection timeout to gitlab.internal.company.com.
3. Tested connectivity: curl -v https://gitlab.internal.company.com — connection    refused, confirming firewall block.
4. Reviewed nftables rules: nft list ruleset — confirmed the outbound ACCEPT rule    for port 443 was missing after the package update reset the ruleset.
5. Re-added the required outbound rule on all 8 runner VMs:
   nft add rule inet filter output tcp dport 443 accept.
6. Persisted the rule: nft list ruleset > /etc/nftables.conf.
7. Restarted gitlab-runner: systemctl restart gitlab-runner.
8. Confirmed runners came online in GitLab admin panel within 60 seconds.
9. Monitored pipeline queue — all 120 pending jobs began processing within 5 min.
10. Updated the Ansible firewall role to include idempotent nftables rules for     runner hosts so future OS patches cannot reset required rules.

PREVENTIVE MEASURES:
Added a post-patch Ansible job in Satellite to validate runner connectivity immediately after any OS patching task.

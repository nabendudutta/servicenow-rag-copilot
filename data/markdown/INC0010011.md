# INC0010011

**Short Description:** Prometheus alertmanager not sending alerts — SMTP relay misconfiguration
**Priority:** 3
**State:** 6

## Description
The Prometheus Alertmanager has silently stopped delivering email and PagerDuty alerts for the past 18 hours. On-call team only discovered the issue when manually checking dashboards. Multiple critical alerts for CPU, memory, and disk thresholds were missed during this window.

## Resolution
ROOT CAUSE ANALYSIS:
A network team change rotated the internal SMTP relay server from smtp-relay-01.internal to smtp-relay-02.internal without updating the Alertmanager configuration. Additionally, the PagerDuty integration key was regenerated during a PagerDuty account audit, invalidating the existing key in Alertmanager config.

RESOLUTION STEPS TAKEN:
1. Checked Alertmanager UI at http://alertmanager:9093 — showed a backlog of 47    firing alerts with 'error sending notification' status.
2. Reviewed Alertmanager logs:
   kubectl logs -n monitoring deployment/alertmanager | grep 'error'.
3. Found two distinct errors: SMTP 'connection refused' to old relay, and    PagerDuty '403 Forbidden' due to invalid integration key.
4. Retrieved the new SMTP relay hostname from the network team's change record.
5. Retrieved the new PagerDuty integration key from the PagerDuty service settings.
6. Updated the Alertmanager ConfigMap in Kubernetes:
   kubectl edit configmap alertmanager-config -n monitoring.
7. Changed smtp_smarthost to smtp-relay-02.internal:25 and updated the    pagerduty_configs service_key field.
8. Reloaded Alertmanager config without restart by sending SIGHUP:
   curl -X POST http://alertmanager:9093/-/reload.
9. Sent a test alert using amtool and confirmed email and PagerDuty delivery.
10. Reviewed all 47 missed alerts manually, triaged any that required action, and     confirmed no production incidents were missed during the outage window.

PREVENTIVE MEASURES:
Added a synthetic alerting canary — a always-firing 'watchdog' alert that pages on-call if no heartbeat alert is received within 5 minutes, ensuring alerting pipeline health is continuously validated.

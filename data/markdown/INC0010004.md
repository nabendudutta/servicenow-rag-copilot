# INC0010004

**Short Description:** OpenShift pods stuck in CrashLoopBackOff after namespace resource quota exceeded
**Priority:** 1
**State:** 7

## Description
Multiple pods in the 'payments-prod' namespace on OCP-CLUSTER-PROD are stuck in CrashLoopBackOff state. The payments processing service is unavailable, impacting all customer transactions. Alert triggered at 02:14 UTC.

## Resolution
ROOT CAUSE ANALYSIS:
The namespace resource quota for CPU requests was set to 8 cores. A recent Helm chart update accidentally doubled the CPU requests per pod replica from 500m to 1000m, causing new pods to fail scheduling and existing pods to be evicted.

RESOLUTION STEPS TAKEN:
1. Identified failing pods: oc get pods -n payments-prod | grep CrashLoopBackOff.
2. Described the resource quota: oc describe resourcequota -n payments-prod to    confirm CPU limits were breached.
3. Reviewed recent Helm deployment history:
   helm history payments-service -n payments-prod.
4. Rolled back the Helm release to the previous revision:
   helm rollback payments-service 14 -n payments-prod.
5. Confirmed pods transitioned from CrashLoopBackOff to Running state within 3    minutes of rollback.
6. Verified all payment transactions resumed by checking application logs and    monitoring dashboards in Grafana.
7. Temporarily increased namespace CPU quota to 16 cores as a buffer:
   oc patch resourcequota compute-resources -n payments-prod    --patch '{"spec":{"hard":{"requests.cpu":"16"}}}'.
8. Fixed the Helm chart values file to restore CPU requests to 500m per pod.
9. Deployed the corrected chart through the proper CI pipeline with peer review.
10. Restored namespace quota back to 8 cores after confirming stable operation     for 2 hours post-deployment.

PREVENTIVE MEASURES:
Added a pre-deployment OPA/Gatekeeper policy to reject Helm values where CPU requests per pod exceed a defined threshold.
